# StorySitting Operating System

This document is the operating doctrine for StorySitting. It exists so future development does not restart from scratch every time the founder and GPT shift into a new work session.

## Current Company Mission

StorySitting helps families gather scattered memories, photos, voices, questions, places, objects, recipes, and interview material into finished private Story Capsules.

## Current Wedge

The first sellable wedge is the Homeplace Story Capsule:

> Before the family home, farm, or meaningful place changes hands, preserve the stories that happened there.

This wedge is sharper than generic family-story preservation because it has a buyer, a deadline, a place, emotional urgency, and a concrete deliverable.

## Product Lock

StorySitting is not primarily a SaaS product, photo product, AI transcription tool, memoir agency, or generic archive.

StorySitting is a guided production system for finishing family Story Capsules.

## Core Pipeline

1. Open Story Room
2. Gather family material
3. Review and organize
4. Create Memory Cards
5. Build Story Map
6. Guide session
7. Produce Capsule
8. Deliver Capsule

Every feature must advance one of these stages.

## Founder Role

The founder owns:

- vision
- taste
- final product direction
- market sensing
- offer prioritization
- real customer judgment
- revenue pressure

The founder should not be forced into constant technical decision-making, file replacement, or manual production chaos.

## GPT / Mission Control Role

GPT owns:

- system architecture
- implementation sequencing
- product workflow design
- UI copy clarity
- production logic
- launch cadence
- reusable component strategy
- debugging and stabilization

GPT should not keep branching strategy mid-build or turning every change into a new website concept.

## Development Modes

### Strategy Mode

Used to decide what matters. No code changes.

### Build Mode

Used to implement one coherent improvement batch. No strategy expansion mid-flight.

### Repair Mode

Used when build or user flow is broken. No feature work until the break is repaired.

### Design Mode

Used only when the product flow is stable and assets are ready.

## Mission Packet Format

Every build mission should define:

- Objective
- Files affected
- Expected result
- Test path
- Stop condition

Example:

Objective: Improve Story Room guidance.
Files affected: Story Room page, workflow component, CSS.
Expected result: owner knows the current phase and next action.
Test path: /story-rooms/[id]
Stop condition: page clearly answers where am I, what next, and what done looks like.

## SpaceX-Style Launch Cadence

1. Define mission
2. Freeze scope
3. Build coherent patch
4. Verify compile/deploy
5. Test one path
6. Capture telemetry/friction
7. Start next mission

No scattered micro-edits unless repairing a specific failure.

## Current Strategic Priority

The next major proof target is one complete canonical Homeplace Story Capsule. That sample becomes sales proof, partner proof, workflow proof, and the model for production quality.

## Decision Filter

Before adding anything, ask:

1. Does this help a family finish a Capsule?
2. Does this reduce founder/operator confusion?
3. Does this move a project to the next production phase?
4. Does this improve the Homeplace wedge?
5. Does this create reusable leverage for future builds?

If not, it waits.
